# KMS - Key Management Service

## Encryption Fundamentals

### Encryption Types

**What:** Methods to scramble data so only authorized people can read it.

**Why we use it:** Protect sensitive data if it's stolen or intercepted.

**How it works:**

```
Symmetric (Same key encrypts AND decrypts):
├── Example: AES-256
├── Speed: Very fast (< 1ms)
├── Key management: Simpler (one key)
└── Use for: Data encryption at rest

Asymmetric (Two keys: public + private):
├── Public key: Anyone can use to encrypt
├── Private key: Only owner can decrypt
├── Speed: Slower but needed for signatures
├── Use for: Key exchange, digital signatures

AWS strategy:
├── Symmetric (KMS) for fast data encryption
├── Asymmetric (RSA) for key sharing
```

**Simple example:**
```
Encrypt a file using AWS KMS:
1. Your app asks KMS: "Encrypt this file"
2. KMS encrypts securely
3. File saved encrypted on disk
4. Later: App asks KMS "Decrypt this file"
5. KMS verifies permissions, decrypts
6. App gets readable file

Security:
├── Even if disk stolen, data is unreadable (encrypted)
├── Only people with KMS permissions can decrypt
```

## KMS Master Keys

### Customer Managed Keys

You control:

```
AWS KMS managed:
├── Storage
├── Rotation
├── Permissions
└── Audit trail

You manage:
├── Who can use
├── Key policy
└── Deletion

Cost: $1/month per key
```

### AWS Managed Keys

AWS controls:

```
Created automatically by AWS:
├── S3, EBS, RDS, etc.
├── AWS manages rotation
└── You can't delete

Cost: Free
```

### Key Rotation

```
Rotate master key:
├── Automatic: Every 365 days
├── Manual: Anytime

Process:
├── New key version created
├── Old version still works (for decryption)
├── New encryptions use new key
└── Old data: Still decryptable

Benefits:
└── Limits cryptographic impact of key compromise
```

## Data Encryption

### Envelope Encryption

```
Data Encryption Key (DEK):
├── Generated by KMS
├── Encrypts large data
└── (Fast, local)

Master Key:
├── Encrypts DEK
├── Stored in KMS
└── Never leaves KMS

Workflow:
1. Get encrypted DEK from KMS
2. Decrypt DEK locally
3. Use DEK to encrypt data
4. Store encrypted data (KMS)

Protection:
└── If data stolen, DEK encrypted (need KMS)
```

### At-Rest Vs. In-Transit

```
At-Rest (data on disk):
├── EBS encryption
├── S3 default encryption
├── RDS encryption
└── KMS manages keys

In-Transit (data over network):
├── SSL/TLS (automatic)
├── VPN for on-premises
├── PrivateLink for internal
```

## KMS Operations

### Generate Data Key

```
bash
aws kms generate-data-key \
  --key-id arn:aws:kms:us-east-1:123456789012:key/12345678-1234-1234-1234-123456789012 \
  --key-spec AES_256

Result:
{
  "Plaintext": "base64-encoded-256-bit-key",
  "KeyId": "arn:aws:kms:...",
  "CiphertextBlob": "encrypted-key"
}

Usage:
1. Use Plaintext for encryption locally
2. Store CiphertextBlob with encrypted data
3. Later: KMS decrypts CiphertextBlob to recover key
```

### Encrypt/Decrypt

```
All operations < 4KB:
aws kms encrypt \
  --key-id alias/my-key \
  --plaintext "secret data"

aws kms decrypt \
  --ciphertext-blob fileb://encrypted.bin

For large data:
└── Use GenerateDataKey instead
```

## KMS Key Policies

Permission model:

```
Example policy (allow IAM role):
{
  "Sid": "Allow Lambda to use key",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::123456789012:role/lambda-role"
  },
  "Action": [
    "kms:Decrypt",
    "kms:GenerateDataKey"
  ],
  "Resource": "*"
}
```

## Use Cases

### Encrypt S3 Objects

```
Automatic encryption:
S3 bucket → Default encryption → KMS key

Every upload:
└── Automatically encrypted with KMS key

Client-side encryption:
1. Get DEK from KMS
2. Encrypt file locally
3. Upload encrypted file to S3 (key stored)
4. KMS never sees file data
```

### Encrypt RDS Backups

```
RDS encryption:
├── Storage encrypted with KMS
├── Backups automatically encrypted
├── Cross-region backup: Can use different key

Recovery:
└── Restore database (auto-decrypts with key)
```

### Secrets Manager

Store secrets safely:

```
Store in Secrets Manager:
├── Database password
├── API keys
├── OAuth tokens

How:
1. Encrypt secret value with KMS
2. Store encrypted secret
3. Application requests secret
4. Secrets Manager decrypts (with KMS)
5. Return plaintext to application

Rotation:
└── Automatic secret rotation (Lambda)
```

## KMS vs. Secrets Manager

```
KMS (lower level):
├── Encrypt/decrypt data
├── Generate encryption keys
├── Cost: $1/month per key
└── More control

Secrets Manager (higher level):
├── Store secret values
├── Automatic rotation
├── Audit trail
└── Cost: $0.40/secret/month
```

## Regional Replication

```
Key in us-east-1 not available in eu-west-1
└── Must replicate key to other regions

Replicate:
├── Create replica in eu-west-1
├── Both keys managed independently
├── Same key material

Use for:
├── Multi-region applications
├── Disaster recovery
└── Data residency requirements
```

## ⚠️ Common Mistakes

❌ **Sharing plaintext keys**
→ Use KMS for key management

❌ **Not rotating keys**
→ Enable automatic rotation

❌ **Over-provisioning key permissions**
→ Least privilege principle

❌ **Forgetting key policy**
→ Default policy blocks all (fix before deploy)

❌ **Regional limitations**
→ Replicate key for multi-region
